# HyDE 假设文档嵌入与高级检索策略

## 核心定义

HyDE（Hypothetical Document Embeddings）由 Luyu Gao 等人于 2022 年提出。核心思想：不直接用用户原始问题检索，而是先用 LLM 生成一篇"假设答案文档"，再用假设答案的向量去检索真实知识库。

## 解决的问题

用户查询往往简短、模糊、口语化（"好玩吗？"），而知识库文档是正式、详细的（"景区评分 4.8 分，游客反馈极佳"）。这种语义鸿沟导致直接检索质量下降。

## 工作流程

```
用户查询（"好玩吗？"）
  → [LLM 生成假设答案]
  → "这个景区非常好玩，有很多娱乐设施..."
  → [向量化假设答案]
  → [向量数据库检索]
  → [注入真实文档到 Prompt]
  → [LLM 生成最终回答]
```

## 适用场景

- 查询模糊、口语化（C 端用户场景）
- 查询与文档风格差异大

## 不适用场景

- 查询已足够精确（如技术文档查询）
- 对延迟和成本敏感的场景

## 关键特点

- **不产生幻觉污染**：假设答案仅用于检索阶段，最终注入 Prompt 的是真实文档
- **可结合路由机制**：根据查询长度和模糊度自动决定是否启用 HyDE

## 相关检索策略对比

| 策略 | 原理 | 额外 LLM 调用 | 适用场景 |
|------|------|:---:|------|
| HyDE | 生成假答案引导检索 | 1次 | 查询模糊 |
| Query Rewriting | 改写查询语句 | 1次 | 查询口语化 |
| RAG Fusion | 多查询融合 | 1次 | 多角度覆盖 |
| Reranker | 二次精排 | 0次 | 检索结果噪音大 |
